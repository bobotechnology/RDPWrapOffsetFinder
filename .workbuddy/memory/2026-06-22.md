## 2026-06-22 22:22 — 修复 Ctrl+C 复制带装饰行（第二次，彻底修复）

- **根因 1**: `tag_ranges` 返回的索引格式不可靠，任何字符串比较都可能失败
- **根因 2**: Text 是 disabled 状态，Windows 上 Ctrl+C 可能走 `<<Copy>>` 虚拟事件而非 `<Control-c>`
- **修复**:
  1. 不比索引格式，直接比较选中文本内容 (`selected.rstrip("\n") == full.rstrip("\n")`)
  2. 同时绑定 `<<Copy>>` 虚拟事件（Windows Text widget 标准复制事件）
  3. 函数名从 `_on_ctrl_c` 改为 `_on_copy`
- 重新构建 exe 验证通过

## 2026-06-22 23:00 — 修复 Dev Log 日志被截断（上下文窗口效应）

- **根因**: `_poll_queue` 每次只处理一条消息就 return 并 reschedule（40ms for log, 80ms for status）。分析线程毫秒级塞入几十条日志，GUI 慢慢一条一条取，取到 `result` 消息就停止轮询，队列里堆着的未取日志全丢
- **修复**: 改为每次 poll 排空整个队列——非终态消息（log/status）处理完继续循环，只有终态消息（result/error/cancelled）才 return 停止轮询
- 55 tests 全绿，两个 exe 重新构建成功

## 2026-06-22 23:13 — 修复 _emit_patch_result 日志前缀 bug（重构丢失）

- **根因**: 重构时 `_emit_patch_result` 用 `label`（如 "SingleUserPatch"）同时做日志前缀和 INI 行搜索前缀，但实际 INI 行前缀是 "SingleUser"（去掉 "Patch"）。`f"{label}Offset.{arch}="` 搜索 `SingleUserPatchOffset.x64=` 永远匹配不到实际的 `SingleUserOffset.x64=`
- **后果**: SingleUserPatch/DefPolicyPatch/LocalOnlyPatch 的 offset/code 详情行和反汇编上下文全部丢失，只剩 "found" 一行
- **修复**: `line_prefix = label[:-5] if label.endswith("Patch") else label`，搜索用 `line_prefix`，日志用 `label`
- 验证: 76 条日志消息全部恢复（含 offset/code 行 + disasm context）
- 55 tests 全绿，两个 exe 重新构建成功
