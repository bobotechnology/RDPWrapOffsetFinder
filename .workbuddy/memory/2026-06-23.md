## DefPolicyPatch Path 2 偏移修复

**问题**: `patches.py` 中 `def_policy_patch` 的 Path 2（双寄存器 CMP 模式）偏移计算错误。
- 原代码: `off_rva = int(insn.ip - ctx.image_base)` → 指向 0x638 MOV 地址
- 修复后: `off_rva = int(insn.next_ip - ctx.image_base)` → 指向 0x63C MOV 地址

**原因**: patch 码 `CDefPolicy_Query_r9d_rdi_jmp` 是 12 字节（`mov [rdi+0x638],0x100` 10B + `jmp short` 2B），需要精确覆盖: 0x63C MOV (7B) + CMP (3B) + JNE (2B) = 12B。0x638 MOV 保留（读旧值无害）。

**验证**: termsrv.dll 10.0.28000.2307 (x64) 输出从 `A0956` 改为 `A095D`，与 Ghidra 分析报告一致。56 个测试全绿。

**其他修复**:
- `_asmharness.py`: `mov_r64_r64` 函数的 `Code.MOV_R64_R64`（不存在）改为 `Code.MOV_RM64_R64`
- 新增测试 `test_two_mov_cmp_reg_jne_offset_after_638_mov` 验证 Path 2 偏移

## DefPolicyPatch Path 2 偏移修复（第二轮 — 反序 MOV）

**问题**: 上一轮修复用 `insn.next_ip` 假设 0x638 MOV 总在 0x63C MOV 之前。但 termsrv.dll 10.0.26100.8521 中顺序相反（0x63C 先，0x638 后），导致 `insn.next_ip` 指向 CMP 而非第二个 MOV。

**修复**: 改用 `insns[k-1].ip`（CMP 前一条指令的地址），不依赖 MOV 顺序。这总是第二条检查 MOV（被 patch 覆盖的那条），第一条 MOV 保留。

**验证**:
- 10.0.26100.8521: `9C544`(CMP, 错) → `9C53D`(0x638 MOV, 对) ✓
- 10.0.28000.2307: `A095D`(0x63C MOV, 对) — 未回归 ✓
- 57 个测试全绿（新增 `test_two_mov_cmp_reg_jne_reversed_order`）

**重新打包**: console 9.3MB + GUI 12.4MB，两个 DLL 均验证通过
